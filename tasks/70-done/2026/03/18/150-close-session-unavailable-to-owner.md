# close_session Tool Unavailable to Owning Session

**Type:** Bug
**Priority:** 150 (High)

## Observed Behavior

During manual testing, Scout (SID 2) received a DM asking it to close its session. Scout reported: "`close_session` tool is currently disabled. Cannot proceed."

The overseer (SID 1) was able to close Scout's session by calling `close_session` with Scout's SID and PIN from a different MCP client instance.

## Expected Behavior

A session should be able to close itself by calling `close_session` with its own SID and PIN.

## Investigation Needed

This could be:

1. **Server-side:** The tool registration or auth middleware somehow prevents a session from calling `close_session` on itself
2. **Client-side:** Scout's MCP client didn't have `close_session` in its available tools list (filtered by client config)
3. **Tool listing:** The `tools/list` response might not include `close_session` under certain conditions

## Completion

**Root cause: Client-side — not a server bug.**

### Investigation Findings

1. **`src/server.ts` line 163**: `registerCloseSession(server)` is called unconditionally — no conditional registration.
2. **`src/tools/close_session.ts`**: Uses `checkAuth(sid, pin)` from `session-auth.ts`. No self-close restriction exists; any valid SID+PIN pair is accepted.
3. **MCP SDK v1.27.1**: `registerTool()` creates tools with `enabled: true` by default. We never call `.disable()` anywhere in the codebase (confirmed via grep). Tools can only become disabled via an explicit `.disable()` call — which we never make.
4. **Conclusion**: The server-side `close_session` tool is always registered, always enabled, and `checkAuth` allows a session to close itself with its own credentials. There is **no server-side restriction** on self-close.

### Root Cause

The phrase "close_session tool is currently disabled" was generated by Scout's AI model (client-side reasoning), **not** returned as an error from the MCP server SDK. The MCP SDK error for a disabled tool is `Tool close_session disabled` — this was not in the reported output. Scout (or its model) declined for reasons internal to the client.

Possible client-side explanations:
- Scout's system prompt or tool restrictions filtered out `close_session`
- Scout's model hallucinated a restriction based on the tool description or prior context
- The MCP client config the operator used for Scout excluded `close_session` from the tool list

### Changes Made

No production code changes — no server bug exists.

**Regression test added** (`src/tools/close_session.test.ts`): "a session can self-close using its own credentials (no server-side restriction)" — documents and guards that self-close works correctly server-side.
